# 8. 외부 연동 현실성 검토

## 8-1. 카카오 선물하기 직접 연동

### 결론: ❌ 사실상 불가능

| 항목 | 현실 |
|------|------|
| **공개 API** | 카카오 선물하기에는 외부 앱이 사용할 수 있는 공개 API가 없음 |
| **파트너십** | 카카오 커머스 파트너 계약 필요. 개인/소규모 팀은 계약 자체가 어려움 |
| **자동 구매/전송** | 사용자 대신 선물을 자동 구매·발송하는 것은 카카오 정책상 불가 |
| **결제 대행** | 앱 내에서 카카오 선물하기 결제를 중개하면 PG사 등록, 전자상거래법 준수 등 법적 요건 발생 |
| **보안 리스크** | 사용자의 카카오 계정 인증 정보를 저장·사용하는 것은 보안상 매우 위험 |

### 법적 경고
- 타인 대신 결제를 진행하는 기능은 **전자금융거래법**, **전자상거래법** 위반 소지
- 카카오 계정 자동 로그인·자동 결제는 카카오 이용약관 위반
- 개인정보(결제 정보) 저장 시 **개인정보보호법** 준수 의무 (ISMS 인증 등)

## 8-2. 기프티콘/상품권 코드 등록 방식

### 결론: ⚠️ 가능하지만 주의 필요

| 항목 | 현실 |
|------|------|
| **구현 방식** | 사용자가 외부에서 구매한 기프티콘 바코드/코드를 앱에 텍스트 또는 이미지로 등록 |
| **난이도** | 쉬움 (텍스트 저장 + 이미지 업로드) |
| **장점** | 실제 보상의 "증거"가 앱에 남음 |
| **단점** | 코드 유출 위험, 앱이 코드를 저장하면 보안 책임 발생 |
| **법적 이슈** | 기프티콘 코드는 금전적 가치가 있으므로, 앱 서버에 저장 시 보안 수준 요구됨 |

### 권장 방식
- 코드 자체를 서버에 저장하지 말고, **사용자 기기에만 로컬 저장** (암호화)
- 또는 코드 등록 기능은 빼고, **인증 사진 업로드**만 지원
- MVP에서는 이 기능을 제외하는 것을 강력 권장

## 8-3. 외부 링크 저장 방식

### 결론: ✅ 가능하고 권장

| 항목 | 현실 |
|------|------|
| **구현 방식** | 보상 설정 시 URL 입력 필드 제공. 보상 시점에 링크를 앱 내 웹뷰 또는 외부 브라우저로 열기 |
| **사용 예시** | 카카오 선물하기 상품 페이지 URL, 네이버 스마트스토어 링크, 쿠팡 상품 링크 |
| **난이도** | 매우 쉬움 (TEXT 필드 저장 + Linking.openURL) |
| **장점** | 연동 없이도 외부 서비스 활용 가능, 법적 리스크 없음 |
| **단점** | 자동 구매가 아니라 사용자가 직접 결제해야 함 |

### 구현
```typescript
// 보상 설정 시
reward_type: 'link'
reward_data: 'https://gift.kakao.com/product/12345'

// 보상 시점에
Linking.openURL(reward.reward_data)
```

## 8-4. 사용자 직접 업로드 보상 인증 방식

### 결론: ✅ 가장 현실적이고 권장

| 항목 | 현실 |
|------|------|
| **구현 방식** | 보상/패널티 이행 후 인증 사진을 찍어서 업로드 |
| **난이도** | 보통 (이미지 업로드 + 스토리지) |
| **장점** | 어떤 종류의 보상이든 유연하게 대응. 법적 리스크 없음 |
| **단점** | 보상 이행을 강제할 수 없음 (자발적 인증) |
| **UX** | "보상 완료" 버튼 → 카메라/갤러리에서 사진 선택 → 업로드 → 상대방에게 알림 |

## 8-5. 앱 내 약속형 리워드 방식

### 결론: ✅ MVP에 가장 적합

| 항목 | 현실 |
|------|------|
| **구현 방식** | 보상을 텍스트로만 기록. "청담 오마카세", "설거지 1주일 면제" 등 |
| **난이도** | 매우 쉬움 (텍스트 필드 1개) |
| **장점** | 구현 단순, 법적 리스크 제로, 사용자 자유도 높음 |
| **단점** | 이행 강제 불가 (앱은 "약속 기록 도구"일 뿐) |
| **이행 확인** | "완료" 버튼으로 상태 변경. 사진 인증은 선택. |

## 최종 권장 전략

### MVP (1차 출시)

| 보상 방식 | 포함 여부 | 이유 |
|----------|----------|------|
| 약속형 텍스트 | ✅ 필수 | 가장 단순하고 유연 |
| 외부 링크 저장 | ✅ 필수 | 쉬운 구현으로 확장성 확보 |
| 인증 사진 업로드 | ⬜ 2차 | 이미지 스토리지 필요, 보통 난이도 |
| 기프티콘 코드 | ❌ 제외 | 보안 리스크, 법적 이슈 |
| 카카오 선물하기 연동 | ❌ 제외 | 기술적/법적으로 불가 |

### 장기적으로 가능한 확장

| 방식 | 조건 |
|------|------|
| 기프티콘 API 연동 (기프팅 등) | 기프티콘 유통 플랫폼과 B2B 계약 필요. 규모가 커진 후 검토 |
| 자체 포인트/코인 | 결제 시스템 도입 시 전자금융업 등록 필요. 과도한 복잡성 |
| 기프티쇼 비즈 API | KT 기프티쇼 비즈 API는 B2B 계약 후 사용 가능. 소량 발송은 단가 높음 |

## 법적 체크리스트

| 항목 | 해당 여부 | 설명 |
|------|----------|------|
| 개인정보처리방침 | ✅ 필수 | 닉네임, 프로필 사진, 푸시 토큰 수집 → 방침 공개 필요 |
| 이용약관 | ✅ 필수 | 서비스 이용 조건 명시 |
| 통신판매업 신고 | ❌ 불필요 | 앱 내 직접 결제/판매 없으므로 (광고만) |
| 전자금융업 등록 | ❌ 불필요 | 포인트/결제 기능 없으므로 |
| ISMS 인증 | ❌ 불필요 | 매출/이용자 수 기준 미달 시 |
| 앱스토어 심사 | ✅ 필수 | 애플: 앱 내 결제 없으면 OK. 광고 SDK 사용 시 ATT 동의 팝업 필수 |
| 아동 보호 | ⚠️ 검토 | 부모-자녀 사용 시 14세 미만 이용자 → 법정대리인 동의 필요 |
