# 12. 개발 난이도와 리스크

## 기능별 난이도 분류

### 쉬움 (1~2일, 막힐 가능성 낮음)

| 기능 | 이유 | 주의점 |
|------|------|--------|
| 프로필 설정 (닉네임) | 단순 CRUD | - |
| 스티커판 수락/취소 | 상태값 UPDATE 1건 | - |
| 스티커 메모 | 텍스트 필드 추가 | - |
| 진행률 표시 | 클라이언트 계산 (current/target) | - |
| 보상/패널티 완료 버튼 | 상태 UPDATE + 알림 | - |
| 히스토리 목록 | SELECT 쿼리 + 리스트 렌더링 | 페이지네이션 고려 |
| 기본 설정 화면 | 리스트 UI + 간단한 UPDATE | - |
| 외부 링크 보상 | URL 저장 + Linking.openURL | - |
| 기본 스티커 이미지 | 에셋 번들에 포함 | 디자인 리소스 필요 |
| 상대방 호칭 설정 | relationships 필드 UPDATE | - |

### 보통 (2~4일, 약간 막힐 수 있음)

| 기능 | 어디서 막힐 수 있는지 | 해결 방법 |
|------|---------------------|----------|
| **소셜 로그인 (카카오)** | 카카오 개발자 콘솔 설정이 복잡. 리다이렉트 URI, 앱 키 설정, Supabase Auth Provider 연결 | Supabase 공식 가이드 참고. 카카오 비즈니스 앱 등록 필요 |
| **소셜 로그인 (구글)** | OAuth 클라이언트 ID 설정, SHA-1 키 등록 (Android) | EAS Build에서 자동 서명 관리 |
| **초대코드 관계 연동** | 코드 충돌 방지, 트랜잭션 처리, 에러 케이스 핸들링 | 6자리 = 36^6 ≈ 22억 조합. 충돌 확률 극히 낮음. Edge Function에서 트랜잭션 처리 |
| **스티커판 생성 폼** | 여러 필드 검증, 날짜 선택 UI, 상태 관리 | React Hook Form + Zod 검증 |
| **스티커 부여 로직** | 동시 요청 시 current_count 정합성 | PostgreSQL의 SELECT FOR UPDATE 또는 unique constraint로 방지 |
| **배너 광고 (AdMob)** | Expo와 AdMob 연동 시 Config Plugin 설정, 테스트/실제 광고 ID 전환 | react-native-google-mobile-ads 공식 문서 참고. 개발 모드에서 테스트 ID 사용 |
| **커스텀 스티커 업로드** | 이미지 리사이즈, 용량 제한, Supabase Storage 업로드 | expo-image-picker로 선택 + 리사이즈 후 업로드. 500KB 제한 |
| **보상 인증 사진** | 위와 동일 (이미지 업로드) | - |
| **관계 해제** | 진행 중 보드 일괄 종료, 양방향 relationship 처리 | Edge Function에서 트랜잭션 처리 |

### 어려움 (4일+, 상당히 막힐 수 있음)

| 기능 | 왜 어려운지 | 구체적 장벽 | 대안/해결 |
|------|-----------|-----------|----------|
| **푸시 알림 전체 구현** | 토큰 관리, 플랫폼별 설정, 서버→클라이언트 전송 체인이 길음 | iOS: APNs 인증서 또는 Key 설정 필요. Android: FCM 설정. Expo Push Service 사용 시 토큰 저장/갱신 로직 필요 | Expo Notifications 사용하면 APNs/FCM 추상화됨. 그래도 초기 설정은 복잡 |
| **기간 만료 자동 처리 (Cron)** | pg_cron 설정, 실패 시 재시도, 알림 연동 | Supabase 무료 플랜에서 pg_cron 사용 가능하나 설정이 SQL 기반. 만료 시 패널티 활성화 + 알림 발송까지 연쇄 처리 필요 | 대안: 클라이언트에서 앱 열 때 만료 체크 (정확도 낮지만 MVP에서는 가능) |
| **애플 로그인** | Apple Developer 계정 ($99/년) 필요, Sign in with Apple 서비스 ID 설정 | Supabase Auth에서 Apple Provider 설정 + App ID, Service ID, Key 등록. 리다이렉트 URL 설정이 까다로움 | iOS 출시 시 필수이므로 피할 수 없음. 구글/카카오 먼저 구현 후 추가 |
| **SNS 공유 카드 생성** | 서버사이드 이미지 렌더링 필요 | HTML/Canvas → 이미지 변환을 서버에서 해야 함. Edge Function에서 Puppeteer 등 사용 어려움 | 대안: react-native-view-shot으로 클라이언트에서 스크린샷 생성 → Share API |
| **홈 화면 위젯** | 네이티브 모듈 직접 작성 필요 | iOS: WidgetKit (Swift), Android: App Widget (Kotlin). Expo에서는 Config Plugin으로 네이티브 코드 주입 필요 | 대안: 성장 후 네이티브 개발자 투입. 또는 expo-widget 커뮤니티 라이브러리 검토 |
| **다중 관계/그룹 지원** | DB 구조 변경 + 권한 모델 재설계 필요 | 1:1 기반으로 설계된 relationships, sticker_boards 구조를 N:M으로 변경해야 함. RLS 정책 복잡도 급증 | MVP에서는 1:1만 지원. 구조 변경은 대규모 마이그레이션 필요 |

## 리스크 매트릭스

```
        높은 영향
           │
  앱스토어  │  소셜로그인    푸시알림
  심사 반려 │  연동 실패     장애
           │
───────────┼──────────── 높은 발생확률
           │
  위젯     │  동시 스티커
  구현 실패 │  부여 충돌
           │
        낮은 영향
```

## 주요 리스크 대응 계획

| 리스크 | 확률 | 영향 | 대응 |
|--------|------|------|------|
| 카카오 로그인 설정 실패 | 중 | 높음 | 구글 로그인을 백업으로 먼저 구현. 카카오 설정은 공식 문서 + Supabase 가이드 병행 |
| 앱스토어 심사 반려 | 중 | 높음 | 개인정보처리방침, ATT 팝업, 애플 로그인 미리 준비. 첫 심사는 여유 있게 1주 전 제출 |
| 푸시 알림 작동 안 함 | 중 | 중 | MVP에서는 인앱 알림(notifications 테이블 조회)으로 대체 가능. 푸시는 2차에서 안정화 |
| Supabase 무료 플랜 한도 초과 | 낮 | 중 | 초기 사용자 500명 수준에서는 한도 내. Pro 플랜($25/월)으로 업그레이드 준비 |
| 동시 스티커 부여 시 데이터 불일치 | 낮 | 낮 | giver가 1명이므로 동시 요청 가능성 매우 낮음. 안전장치로 sequence unique constraint 추가 |
